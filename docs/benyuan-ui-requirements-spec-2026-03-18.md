- 文档版本：v1.0
- 更新时间：2026-03-18
- 适用对象：产品、设计、前端、Prompt / Agent 协同
- 依据来源：

  - 参考图所体现的原生移动端 onboarding / app 风格

---

## 1. 文档目标

这份文档不是单纯的视觉说明，而是 **本源 UI 的统一需求基线**。它要解决三个问题：

1. 当前主流程哪些页面属于正式产品路径，哪些属于 internal / debug。
2. 每个页面当前已经承载了哪些板块和状态，哪些必须保留，哪些必须移出主路径。
3. 最终本源应该以什么样的视觉、信息密度、交互语言和状态闭环落地。

结论先行：

- **本源主流程** 必须收成一条单焦点、移动端优先、原生 app 感极强的路径：
  - `/collect`
  - `/processing/benyuan`
  - `/theater`
  - `/constellation`
- **主流程默认不可见**：测试包、freeze demo、runtime/provider、debug 控制、工程状态说明、站点导航、工具页语气。
- **内部页继续保留**：`/lab/status`、`/lab/native-handoff` 及其他 lab / debug 页面，但它们只承担内部复核和调试，不进入用户心智。

## 2. 总体产品判断

### 2.1 本源不是一个“工作台”产品

用户实际体验的不是后台、看板、控制台，也不是问卷表单。用户进入本源时，应当感受到的是：

- 一个被设计过的、封闭的、连续的心理探索流程
- 一次从“进入 → 回答 → 等待 → 剧场 → 结果”的沉浸式旅程
- 每一屏只有一个主任务，不需要解释太多，也不需要同时处理多块信息

### 2.2 参考图对本源的启发，不是颜色，而是语言

参考图传达的不是蓝紫色，而是以下 UI 语言：

- 极低 chrome
- 粗无衬线大标题
- 大圆角卡片 / pill CTA
- 中心发光 / 氛围渐变
- 强单焦点
- 首屏即情绪，不是首屏即说明
- 底部主操作强承接
- 移动端优先，不像网页后台

本源在借鉴时要做映射，而不是照抄：

- **保留本源黑 / 金 / 灰主题**
- **复刻参考图的版式、层级、节奏、触感**
- **不复刻参考图原始蓝紫色**

## 3. 设计总原则

### 3.1 体验原则

1. 一屏一焦点
2. 先内容，后壳层
3. 先情绪，后解释
4. 先移动端，后桌面兼容
5. 主路径低信息量，内部页高信息量

### 3.2 视觉原则

1. 黑底 + 金色光晕 + 灰阶文本
2. 粗系统无衬线作为主流程第一字体语言
3. 大圆角，不再使用主流程中的硬直角工作台卡片
4. 厚实主 CTA，弱化次级动作
5. 强氛围背景，弱化“章节标题制造氛围”的旧做法

### 3.3 文案原则

1. 文案短，不解释用户“正在做什么”太多
2. 主路径避免工具语气、工程语气、状态看板语气
3. processing / empty / retry 也必须使用同一套克制语言
4. 剧场与结果页允许文学性，但要克制，不能变成长说明书

### 3.4 信息原则

1. 主路径只保留用户决策必需信息
2. 主路径隐藏 runtime / provider / model / request id / telemetry 等工程信息
3. 测试包、demo、debug 路径保留，但只出现在 lab 或显式 debug 模式
4. 完整结果能力保留，但默认折叠，不把第一次阅读做成“长报告”

   - 当前阶段短说明
   - 单条进度轨
3. actions
   - 重试
   - 回上一步
   - 或返回前一步 CTA

### 当前阶段链路（Part 1 → Theater）

1. 收束特征
2. 解析线索
3. 构建剧场

1. 记录选择
2. 精神分析
3. 星图显形

### 本页最终目标

这是一个 **单中心等待页**，不能长成状态看板。

- 单条进度轨

1. 顶部壳：返回 + 细进度

## 8. 交互规范

## 8.1 顶部壳

### 主流程统一规范

- 左：返回
- 中：2px 细进度条
- 右：简短进度文本
- `/collect` 额外允许 A/B/C 微切换

- 主 CTA 永远最强

- 单列优先
- 44px 以上触达面积
- 按压反馈统一
- 选中态只允许金色描边 / 发光提升，不使用多色编码

- 只允许一个上传舞台
- 原生相册 / 拍照为同层副动作
- 缩略图为横向轨道
- 不再长出第二层素材状态面板

## 8.5 空态 / 重试态

- 必须复用主流程沉浸骨架
- 不允许回退到 `SectionTitle + 信息卡 + 大段说明` 的旧占位样式

## 9. 内容与文案需求

## 9.1 主流程文案层级

### 一级：身份与情绪

用于：页面首屏、大标题、结果原型名、场景主句

要求：

- 短
- 直接
- 有情绪，但不过度诗化

### 二级：必要说明

用于：一句承接文案、镜像提示、等待文案

要求：

- 只说用户当前必须知道的事
- 不解释系统内部机制

### 三级：调试与工程信息

用于：lab/debug/internal

要求：

- 只在内部页出现
- 不进入用户主路径

## 9.2 Agent 输出对 UI 的要求

本轮不改 schema，但 UI 端文案和展示逻辑必须与下面要求对齐：

1. 剧场输出：
   - 更像场景句和对话句
   - 不像说明文本
2. 星图 narrative：
   - 更像镜像感短段落
   - 不像测试报告
3. recommendation 原因：
   - 更像“为什么与你共鸣”
   - 不像模板推荐词

## 11.2 主流程禁止显示

- runtime / provider / model / request id
- benchmark / freeze / test pack / demo route
- hover 次数 / hesitation / trait / telemetry
- 工具说明、开发进度、调试入口
- 过量状态摘要
- 网站式 header / 菜单式导航

## 11.3 内部页允许显示

- 调试路径
- benchmark / freeze / shell / handoff 状态
- regression 结果
- 真机验收板
- runtime / debug override

## 12. 响应式与 iPhone shell 要求

### 12.1 iPhone 优先级最高

所有主流程页面设计和验收都必须以 iPhone shell 为最高优先级。

### 12.2 必须保证

1. safe area 正确
2. 底部 sticky CTA 不压 home indicator
3. 非输入区长按不弹系统选词
4. 可复制区继续允许选择
5. 滚动承接顺滑
6. 页面跳转不出现白屏或布局回流突变

### 12.3 桌面端策略

- 只做兼容
- 不单独设计第二套复杂桌面布局
- 不允许桌面端把主流程重新变回工作台信息流

## 14. 最终验收标准

## 14.1 视觉与体验验收

1. 主流程四页不再有工作台感
2. 顶部 chrome 明显弱于当前内容
3. 当前题 / 当前场景 / 当前结果始终是第一视觉焦点
4. 主路径中不再出现开发信息、调试信息、运行时信息
5. 各页间视觉语言一致，不出现风格跳变

## 14.2 页面级验收

### `/collect`

- 默认只看到：返回、细进度、弱 A/B/C、当前题、输入/选项、底部操作
- 上传题不出现工具面板感
- 主 CTA 在 iPhone 下尽量稳定可见

### `/processing/benyuan`

- 任一状态都只有一组中心信息
- 不出现双层进度或重复状态语句

### `/theater`

- 不出现任何 telemetry / trait / instrumentation 信息
- 第一幕不是说明页，而是 hero 开场

### `/constellation`

- 默认阅读长度控制在：首屏 + 3 段短结果 + 1 个折叠入口
- 首屏不能被动作区、工具信息抢戏

### `/lab/status` / `/lab/native-handoff`

- 信息完整
- 层级统一
- 工具态清楚，但不挤进主路径

## 16. 一句话定义本源 UI

**本源不是一个展示很多信息的产品，而是一条被 carefully controlled 的、移动端优先的、单焦点精神探索路径。**

用户需要看到的只有：

- 现在到哪一步了
- 当前这一题 / 这一幕 / 这一段结果是什么
- 下一步该做什么

除此之外的所有信息，都应该退出主路径。
