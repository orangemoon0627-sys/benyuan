# 「本源」Result Page Content Map v0.1

本文件定义 `/report/:sessionId` 页面的内容结构、优先级和文案职责。

## 1. Page Goal

结果页不是“数据展示页”，而是产品价值真正发生的页面。

它需要同时完成四件事：
- 让用户感觉被理解
- 让长文本易读
- 让结果显得有结构而不是散文堆砌
- 让保存/分享变成自然动作

## 2. Content Hierarchy

建议顺序如下：

1. Hero / 精神原型首屏
2. 精神地形总览
3. 三维解读
4. 内在张力
5. 推荐内容
6. 安全提示与边界说明
7. 保存/分享/重测入口

## 3. Section Map

## 3.1 Hero Block

内容：
- 原型名称
- 原型副标题可选
- 一句短引导语
- 轻量 metadata：例如“本次画像基于你的审美、情绪与时间线索整理”

目的：
- 让用户一进入结果页就抓住一个核心意象

不要：
- 一上来塞满大段文字
- 把所有解释都堆到首屏

## 3.2 Overview Block

内容：
- `overview`

目的：
- 建立“整体被理解感”
- 形成情绪上的着陆点

版式要求：
- 适合单独截图
- 宽度不宜过宽
- 段落切分清晰

## 3.3 Dimension Blocks

三个维度分别独立展示：
- 审美语法
- 情感气候
- 时间哲学

每块包含：
- 小标题
- 主解释文案
- 置信度提示（轻量）
- 若有需要，可展示 1-2 个关键词标签，但不要像测试分数卡

目标：
- 让用户看到“具体哪里像我”

## 3.4 Tension Blocks

每组张力包含：
- 张力名称
- 两极关键词
- 描述
- 共处建议

目标：
- 把“为什么你会又这样又那样”说清楚

注意：
- 这是结果页里最容易产生“说中了”的部分
- 但不能写成矛盾学鸡汤

## 3.5 Recommendation Blocks

建议拆成 4 类：
- 哲学建议
- 书
- 音乐
- 小练习

呈现方式：
- 卡片式或小节式
- 每条 recommendation 一屏内能看明白

目标：
- 让用户感到后续还有路可走
- 但不要压过洞察本身

## 3.6 Safety Note Block

默认较轻：
- “这是一份理解性镜像，不构成诊断。”

如果有 flag：
- 改成更明确的支持性文案

目标：
- 让边界清楚
- 不破坏整体氛围

## 3.7 Action Block

包含：
- 保存结果
- 分享结果
- 重新测试
- 稍后查看

目标：
- 不要让行动按钮打断阅读
- 放在阅读结束后的自然位置

## 4. Mobile-First Notes

移动端结果页要注意：
- 首屏只承载原型和一句短引导
- overview 段落不要过长
- dimension 与 tension 卡片要便于滑动阅读
- action block 要固定在较后位置，不抢戏

## 5. Copy Responsibilities by Section

### Hero
负责“抓人”

### Overview
负责“让我觉得被看见”

### Dimensions
负责“让我知道具体哪里像我”

### Tensions
负责“解释我内部的复杂性”

### Recommendations
负责“给我一个可以继续走的方向”

### Safety Note
负责“告诉我这不是定义，也不是诊断”

## 6. Result Page Success Signals

如果结果页设计得好，用户应该会出现以下行为：
- 停留时间长
- 滚动深度高
- 对某段话截图
- 回头重看某一个张力或某一维解读
- 点击保存/分享而不是立刻退出

## 7. Rendering Constraints

页面渲染必须保证：
- 所有区块都能独立缺省，不因某块内容少而崩
- `low_information` 情况下也有完整页面结构
- `self_harm_risk` 情况下自动替换为危机支持优先结构
