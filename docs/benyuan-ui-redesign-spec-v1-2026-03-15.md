# 本源 UI 整体重构规格文档 v1

更新时间：2026-03-15  
适用范围：当前 `beta-2026-03-11-r2` 基线后的 UI / 体验整体大改  
文档目标：为后续整套 UI 大改提供一份可直接执行的规格文档。重点是**整体视觉、结构、交互、文案包装重做**，但**核心测试内容、主链路、数据 contract、回归护栏不变**。

---

## 1. 本轮重构总原则

### 1.1 本轮要做什么

本轮不是修小样式，而是做一次**整体界面层重构**，目标是把当前 beta 从“功能完成、可跑通”提升到：

- 更统一的产品完成态
- 更强的 iPhone app 感
- 更清晰的主流程节奏
- 更稳的黑金极简视觉系统
- 更适合后续长期以 app 形式使用

### 1.2 本轮不做什么

本轮**不重写业务内核**，不动这些内容：

- 核心测试题内容
- 题目 ID、题目分组、题目校验规则
- 上传题的题目位置与核心用途
- Part 1 / Part 2 / Part 3 主数据结构
- 公共 API 路由与返回结构
- benchmark 输出结构
- 本地 storage key / pending key / session key
- native bridge message 名称：`share`、`openExternal`、`pickImages`
- `Web 主流程 + WKWebView iOS shell` 架构

### 1.3 一句话目标

**保留当前已跑通的测试与分析骨架，只把“用户看到、摸到、读到、感受到”的那一层整体重做。**

## 2. 不可变边界

这是本轮所有 UI 重构必须遵守的硬边界。

### 2.1 流程边界

主流程不变：

- `/collect`
- `/processing/benyuan`
- `/theater`
- `/constellation`

流程顺序不变：

- Part 1 采集
- processing 显影
- Part 2 剧场
- processing 显影
- Part 3 星图

### 2.2 测试内容边界

不得改动：

- A / B / C 三组的核心测试内容
- 题目含义
- 核心选项语义
- 上传题核心功能
- 时间分布题逻辑
- 多选题最少选择要求
- 已有 archetype / constellation 生成输入骨架

允许改动：

- 题目的视觉呈现方式
- 单题排版
- 选项卡片样式
- 模块之间的承接方式
- 引导文案
- 状态反馈文案

### 2.3 工程边界

不得改动：

- 公共接口 contract
- benchmark JSON wire shape
- storage keys
- native bridge names
- 现有 freeze / golden / regression 护栏结构

允许改动：

- 页面结构层级
- CSS / UI recipes / primitives
- 页面 copy
- 壳层视觉承接
- 页面内交互状态

## 3. 重构后的产品定位

### 3.1 产品感知目标

重构后的本源，不应该给人“在手机里打开一个网页问卷”的感觉，而应该更像：

- 一个完整的黑金极简精神探索 app
- 一个带仪式感的路径产品
- 一个以结果显影和主观体验为中心的产品

### 3.2 感知关键词

本轮统一的感知关键词固定为：

- 冷静
- 深邃
- 节制
- 神秘
- 仪式感
- 呼吸感
- app 化
- 非网页感

### 3.3 设计判断标准

如果一个页面改完以后仍然像：

- 普通后台页面
- 普通问卷页
- 普通网页导航站
- 普通图文报告页

那就说明这轮重构还不够。

## 4. 全局视觉系统规格

### 4.1 颜色系统

继续保留黑白金主系统，但要进一步统一使用规则。

#### 主色

- 背景主黑：页面主背景
- 深层黑：卡片、分层背景
- 表面黑：高一级内容容器
- 纯白：主标题、关键数字
- 中灰：正文、说明
- 深灰：次要标签、辅助信息
- 金色：唯一强调色

#### 使用规则

- 金色只用于：
  - 当前焦点
  - 当前主操作
  - 当前 active 状态
  - 关键统计或高价值提示
- 不允许在页面里出现第二套主强调色
- 不允许用彩色区分业务模块

### 4.2 字体系统

保持当前方向，但要更严格：

- 大标题：轻字重、强留白
- 正文：克制、清楚、不要太密
- 辅助说明：更轻、更稳
- 元信息：统一为小号 uppercase/等宽感标签

#### 文本层级建议

- 顶层标题：用于每页 hero
- 区块标题：用于核心区块
- 区块导语：用于建立阅读预期
- 正文：用于说明和 narrative
- meta：用于阶段、状态、标签、编号

### 4.3 卡片系统

本轮必须统一卡片系统，不允许各页继续各写各的。

卡片只保留四种语义：

1. **主舞台卡**
   - 页面当前最重要的内容容器
   - 例如当前问题、当前场景、当前结果主摘要

2. **信息卡**
   - 用于状态、说明、摘要、统计

3. **强调卡**
   - 用于当前重点提示、当前阶段承接、风险提醒、推荐入口

4. **交互卡**
   - 用于选项、模块切换、阅读路径、推荐项

### 4.4 间距系统

重构后要统一呼吸节奏：

- 页面级间距：大
- 区块间距：中大
- 卡片内部间距：中
- 标签、按钮与元信息间距：小

原则：

- 页面不能再显得拥挤
- 当前焦点区域必须有明显呼吸感
- 不能用堆密度来表现“高级感”

## 5. 全局交互与 iOS app 感规格

### 5.1 目标

目标不是把 Web 重写成原生，而是让当前 shell 里的 Web 主流程更像 app。

### 5.2 必须满足的感知标准

#### 文本与长按

- 普通标题、段落、按钮文字不可被长按选词
- 输入区、可复制区保留选择能力
- 不出现明显浏览器式复制菜单

#### 点击与按压

- 所有主要按钮、选项卡、模块卡、推荐卡都要有统一按压反馈
- 反馈必须克制，不夸张
- 不能像普通网页按钮那样“只是 hover 换颜色”

#### 顶部承接

- shell 顶部导航要像 app 顶栏，不像网站导航
- 当前页面标题、当前阶段、副说明要明确
- 导航项不宜过多

#### 底部承接

- 底部主操作区要像 app 操作坞
- 尤其在 `/collect`、`/theater`、`/constellation` 中，主操作要稳定
- iPhone 安全区必须考虑进去

#### 滚动手感

- 页面滚动区域要清楚
- sticky 区块不能挡内容
- 大段结果页不能显得像无组织长网页

### 5.3 本轮壳层要求

允许做：

- shell 顶栏重构
- safe area 强化
- 底部操作区统一
- 页面过渡与触感收口

不做：

- 原生页面拆分
- 新 native 页面栈
- 新 bridge message
- 原生持有业务状态

## 6.1 `/collect` 重构规格

### 页面目标

重构后的 `/collect` 必须同时满足：

- 是主入口
- 是采集现场
- 是状态总览
- 是 app 里的“第一层核心工作台”

不能再给用户“内容很多但中心不够明确”的感觉。

### 当前问题

1. 信息很多，但主次还可以更清楚
2. 虽然已经补了承接逻辑，但视觉上“下一步”还不够强
3. 状态区、模块区、问题区、分析入口区之间仍有竞争感
4. 作为 app 首屏时，头部与底部承接仍可以更整体

### 重构目标

#### 目标一：改成“单主舞台 + 三个辅助层”

推荐重构结构：

1. 顶部：页面总态 + 当前阶段
2. 中部主舞台：当前问题
3. 左右/上下辅助：模块、进度、素材状态
4. 底部：主推进区

换句话说：

- **当前问题必须是绝对主角**
- 其他信息必须围绕当前问题服务

#### 目标二：模块切换更清楚

A/B/C 模块要像三个章节，不像三个普通 tab。

要求：

- 当前模块突出
- 已完成模块有明显完成感
- 待开始模块保持克制
- 完成一组进入下一组的承接提示更有仪式感

#### 目标三：上传题体验更 app 化

上传题不是技术动作，而是“把素材交给系统”。

所以要强调：

- 当前上传状态
- 素材已进入链路
- 来源（网页 / 原生相册 / 原生拍照）
- 缩略图确认感

### 页面建议结构

#### A. 顶部总态区

保留内容：

- 总进度
- 当前模块
- 素材入链
- 当前运行状态

但结构上改成：

- 一张主总态卡
- 两到三张次级状态卡
- 少一些并列同权块

#### B. 模块导航区

建议改成更像章节切换的结构：

- A / B / C 每个模块像一个章节标签
- 用完成状态、当前位置、下一步来强调路径感
- 文案上尽量少技术味，多阶段感

#### C. 当前问题区

这是 `/collect` 的绝对主舞台。

必须强化：

- 当前问题标题
- 当前问题说明
- 当前题型的视觉专用布局
- 当前题已完成 / 未完成状态

不同题型要求：

- 单选题：更简洁、更像选择仪式
- 多选题：更像靠近多个倾向，而不是普通复选框
- 上传题：更像“交付素材”而不是“上传文件”
- 分配题：更像权重拉扯，不像表单

#### D. 底部主推进区

必须统一成 app 主操作区。

状态必须覆盖：

- 当前题未完成
- 当前题已完成 -> 进入下一题
- 当前模块已完成 -> 进入下一组
- 三组全部完成 -> 进入剧场

### 文案包装要求

`/collect` 页统一语气：

- 少说“填写、提交、表单”
- 多说“交给、进入、继续、承接、推进、显影”

### 不允许动的内容

- 题目内容
- 题目顺序（除非后续单独定）
- 题目 ID
- 上传接口
- 提交链路

## 6.2 `/processing/benyuan` 重构规格

### 页面目标

要让等待过程更像一场显影，而不是中间页占位。

### 当前问题

1. 链路已经清楚，但视觉氛围还可以更强
2. 处理中的阶段感还可以更“有呼吸”
3. 成功、失败、恢复的视觉语气还可以更统一

### 重构目标

- 强化“显影过程”而不是“系统处理”
- 每个阶段的开始、完成、失败要更易读
- 处理中页面要更适合 iPhone shell 长时间停留

### 必要保留

- 阶段结构
- runtime 信息映射
- checkpoint / resume 逻辑
- 成功后的自动跳转

### 视觉方向

- 更克制的加载节奏
- 更像“正在展开”的过程
- 更少普通加载器的既视感

---

## 6.3 `/theater` 重构规格

### 页面目标

剧场页要比现在更沉浸，但不能牺牲清晰度和可回归性。

### 当前问题

1. 已经有结构，但四幕之间的统一舞台感还不够极致
2. 第二幕、第三幕在视觉上仍偏“内容区块切换”
3. 尾声到星图的情绪承接还能再更完整

### 重构目标

#### 目标一：四幕统一成同一舞台语言

- 幕前说明
- 主场景
- 选择卡
- 状态提示
- 底部推进

都要统一同一套视觉骨架。

#### 目标二：选择行为更有“戏剧感”

- act2 的选择不是普通选项列表
- act3 的镜像回答不是普通题卡
- 重点是“沉浸式对话”感

#### 目标三：系统记录感不能消失

尽管更沉浸，仍要保留：

- 当前章节
- 当前进度
- 已记录数量
- 当前下一拍

但这些信息要更像舞台提示，而不是后台统计。

### 文案包装要求

- 更诗性
- 更慢
- 更少工具味
- 更像在被引导进入下一拍

### 不允许动的内容

- 三幕结构
- choice logs / mirror logs 记录逻辑
- 结束后进入 Part 2 processing 的链路

---

## 6.4 `/constellation` 重构规格

### 页面目标

这是本轮 UI 大改里最关键的一页。

重构后的结果页必须满足：

- 第一眼就有结果完成感
- 不是普通报告页
- 移动端容易扫读
- 长内容不会压垮用户
- 分享与导出更自然

### 当前问题

1. 当前已经比普通报告页强，但仍偏“高级报告页面”
2. 顶部摘要、七维、张力、建议、推荐之间还可以更统一
3. 结果页的文学感和原型气质还可以再拉开
4. 动作区还可以更像 app 的结果操作区

### 重构目标

#### 目标一：先读摘要，再进深读

结果页需要更明确的阅读路径：

1. 原型总感
2. 四个最重要摘要
3. 七维与解释
4. 精神地形 narrative
5. 核心张力
6. 成长建议
7. 推荐内容

#### 目标二：移动端扫读效率更高

要求：

- 每个区块都能独立阅读
- 大段文字要拆层
- 当前聚焦的维度说明要足够清楚
- 不出现“必须通篇读完才知道重点”的问题

#### 目标三：操作区更完整

结果页的操作区要明确分三类：

- 保存 / 导出
- 分享
- 开始新的旅程 / 返回上一步

### 文案包装要求

结果页语气应是：

- 理解你
- 不吓人
- 不像诊断
- 不像鸡汤
- 不是模板报告腔

### 不允许动的内容

- constellation 结果 schema
- 核心字段名
- 建议与推荐的结构层级
- 分享桥接 contract

- 统一文案语气
- 保持工程可见性

### 要求

- 更像产品内部控制台
- 不要太像开发者临时调试页面
- freeze / benchmark / golden / readiness 之间的关系更清楚

## 7. 文案重构规格

### 7.1 本轮文案总方向

文案统一采用三层结构：

#### 第一层：仪式感语言

用于：

- 入口
- 当前阶段
- 页面标题
- 转场提示

关键词：

- 进入
- 展开
- 承接
- 显影
- 路径
- 下一拍
- 星图
- 回响

#### 第二层：解释性语言

用于：

- 区块说明
- 用户理解当前在干什么
- 结果页阅读提示

要求：

- 清楚
- 克制
- 不技术化

#### 第三层：系统语言

用于：

- 状态页
- 内部面板
- 错误态
- 恢复态

要求：

- 准确
- 明确
- 不制造误解

### 7.2 本轮需要统一替换的表达方向

尽量减少：

- 提交
- 表单
- 选项
- 接口
- runtime（直接暴露给用户）
- provider（直接暴露给用户）

尽量改成：

- 进入下一步
- 当前状态
- 当前路径
- 当前分析链路
- 系统正在显影
- 当前来源 / 当前方式

---

## 8. 页面信息架构要求

### 8.1 必须做到“每页只有一个主任务”

每页必须有一个绝对主任务：

- `/collect`：完成当前问题并继续推进
- `/processing/benyuan`：理解当前正在发生什么
- `/theater`：沉浸做出当前选择
- `/constellation`：读懂当前结果并带走一个动作

### 8.2 每页都必须有明确的“当前焦点”

不能出现用户不知道自己当前该看哪里。

### 8.3 每页都要有明确的“下一步”

不能让用户自己猜流程。

## 9. 实施顺序规格

本轮整体大改建议严格按下面顺序执行。

### 第一阶段：共享层先行

先改：

- `src/config/benyuan-ui-recipes.ts`
- `src/components/framework-primitives.tsx`
- `src/app/globals.css`
- `src/components/site-header.tsx`
- `src/components/app-shell.tsx`

目标：

- 先把 token、按钮、卡片、顶部壳层、底部操作感统一好

### 第二阶段：先做 `/collect`

原因：

- 它是入口页
- 承担最多结构重排价值
- 改完后，整套产品的第一印象会立即改善

### 第三阶段：再做 `/theater`

原因：

- 它决定整体体验感和仪式感
- 适合在共享层成熟后继续放大沉浸感

### 第四阶段：最后做 `/constellation`

原因：

- 它的内容最重
- 最适合在前两页语言稳定后统一收束

### 第五阶段：补 `/processing` 与内部页统一

原因：

- 它们更适合在主路径语言稳定后统一口径

## 10. 测试与验收规格

### 10.1 UI 改动后的固定回归

每轮必须至少跑：

- `npm run build`
- `BENYUAN_BASE_URL=http://127.0.0.1:3015 npm run ios:shell:regression`
- `BENYUAN_BASE_URL=http://127.0.0.1:3015 npm run ios:shell:native-smoke`

页面人工回归至少检查：

- `/collect`
- `/processing/benyuan`
- `/theater`
- `/constellation`
- `/lab/status`
- `/lab/native-handoff`

### 10.2 每页验收标准

#### `/collect`

- 当前问题清楚
- 模块承接不迷路
- 上传题确认感强
- 底部主操作稳定

#### `/processing/benyuan`

- 当前阶段清楚
- 中断与恢复语义清楚
- 不像普通 loading

#### `/theater`

- 四幕统一
- 当前拍点清楚
- 主观感受更沉浸

#### `/constellation`

- 先读什么很清楚
- 摘要可一眼抓住
- 移动端扫读顺
- 分享导出入口清楚

### 10.3 本轮失败判定

如果出现以下任一情况，就算本轮 UI 重构不合格：

- 打断主链路
- 影响上传回流
- 影响 processing 跳转
- 影响 theater -> constellation 跳转
- 影响 shell regression / native smoke
- 让页面主任务更不清楚

## 11. 设计稿与实施沟通格式

如果后续你要继续给我方向，最有效的输入方式不是长篇抽象描述，而是按这四类给：

### A. 页面优先级

例如：

- 先做 `/collect`
- 再做 `/theater`
- 最后做 `/constellation`

### B. 风格方向

例如：

- 更像原生 iPhone app
- 更黑更空更冷
- 更强仪式感
- 更强结果完成感
- 更文学，但不矫饰

### C. 明确不能碰的点

例如：

- 不改当前题目内容
- 不改结果结构
- 不改分享逻辑
- 不改 processing 文案主方向

### D. 你觉得最不满意的感知问题

例如：

- 还是太像网页
- 结果页不够高级
- 剧场不够沉浸
- collect 太散

## 12. 最终结论

这轮 UI 整体重构的正确策略不是“推翻重做”，而是：

- **保留已跑通的测试与分析骨架**
- **重做页面结构、视觉节奏、交互触感与文案包装**
- **把当前 beta 收成真正可长期使用的 app 化产品界面**

后续实施主线固定为：

1. 共享视觉系统
2. `/collect` 整体重构
3. `/theater` 沉浸式重构
4. `/constellation` 结果页重构
5. `/processing` 与内部页统一收口

如果按这份规格执行，最终得到的不会是“多了点样式的网页”，而是：

**一套保留当前功能骨架，但感受上更完整、更稳定、更像 iOS app 的本源正式界面。**
